
# DDD v1.0 00: Conventions and Shared Contracts

## 1. Purpose

Defines shared data types, API conventions, error codes, and inter-service contracts used across all DDD documents. All component-level documents reference this document for common structures.

## 2. API Conventions

### 2.1 Transport

All service-to-service calls use **HTTPS + mTLS**. When a service mesh (Istio/Cilium) is deployed, mTLS is enforced transparently at the sidecar layer; application-level URLs inside the mesh may use `http://` (the mesh upgrades to mTLS). URLs that are not protected by a service mesh sidecar (e.g., external cloud APIs) must use `https://`.

### 2.2 Request Format

```
POST /v1/<resource>
Content-Type: application/json
X-Request-ID: <uuid-v4>            # generated by API Gateway; propagated throughout
X-Trace-ID: <uuid-v4>              # same as request_id for v1.0 (one-to-one)
Authorization: Bearer <token>       # only at API Gateway boundary; stripped before internal hops
X-Trusted-Claims: <base64-json>    # verified claims forwarded by API Gateway on internal calls
X-Claims-Sig: <hmac-sha256-hex>    # HMAC-SHA256 signature of the claims JSON body
```

> **Open decision (HLD §11-open-decisions #1)**: The final format and signature verification method for trusted upstream claims has not yet been finalized. The `X-Trusted-Claims` + `X-Claims-Sig` (HMAC-SHA256) design above is the v1.0 working proposal. This must be confirmed before the claims ingestion path is implemented.

### 2.3 Response Format

All responses use `application/json` with the following envelope:

**Success:**
```json
{
  "request_id": "uuid-v4",
  "data": { /* endpoint-specific payload */ }
}
```

**Error:**
```json
{
  "request_id": "uuid-v4",
  "error": {
    "code": "ERR_<DOMAIN>_<REASON>",
    "message": "Human-readable description (safe to expose to caller)",
    "details": {}       // optional; omit on sensitive paths
  }
}
```

### 2.4 HTTP Status Codes

| Status | When |
|--------|------|
| 200 | Success |
| 400 | Bad request (malformed input, missing required fields) |
| 401 | Authentication failed (invalid or missing token) |
| 403 | Authorization denied (valid identity, insufficient permission) |
| 429 | Rate limit exceeded |
| 500 | Internal error (safe message only; no stack traces) |
| 503 | Service temporarily unavailable (circuit open, fail-closed) |

### 2.5 Error Code Registry

| Code | HTTP | Meaning |
|------|------|---------|
| `ERR_AUTH_INVALID_TOKEN` | 401 | Bearer token rejected by API Gateway |
| `ERR_AUTH_MISSING_CLAIMS` | 401 | Required claim fields absent |
| `ERR_AUTH_UNTRUSTED_CLAIMS` | 403 | Claims source not trusted |
| `ERR_AUTH_CLEARANCE_INSUFFICIENT` | 403 | User clearance < required sensitivity level |
| `ERR_GUARD_INJECTION_DETECTED` | 400 | Query Guard blocked for injection risk |
| `ERR_GUARD_ENUMERATION_DETECTED` | 429 | Enumeration pattern detected |
| `ERR_GUARD_RATE_LIMIT` | 429 | Per-user rate limit exceeded |
| `ERR_QUERY_PARSE_FAILED` | 400 | Query Understanding could not parse input |
| `ERR_RETRIEVAL_EMPTY` | 200 | Authorized query returned zero results (not an error, delivered as 200 with empty data) |
| `ERR_RETRIEVAL_FAILED` | 503 | Elasticsearch index unreachable or timed out (infrastructure failure, not an empty result) |
| `ERR_RERANKER_TIMEOUT` | 200 | Reranker timed out; response uses retrieval order |
| `ERR_MODEL_UNAVAILABLE` | 503 | Model Gateway unreachable |
| `ERR_AUDIT_FAILED_CLOSED` | 503 | L2/L3: audit write failed; response withheld (fail-closed) |
| `ERR_ACL_STALE_CACHE` | 503 | acl_version mismatch; recomputation failed |
| `ERR_INTERNAL` | 500 | Unexpected internal error |

## 3. Shared Data Types

### 3.1 UserContext

Produced by the Claims-to-ACL Adapter and passed to all downstream query components.

```typescript
interface UserContext {
  user_id: string;                  // normalized; from upstream identity
  effective_groups: string[];       // deduplicated, sorted group tokens
  effective_clearance: number;      // integer 0–3 (L0=0 … L3=3)
  acl_tokens: string[];             // bounded ≤ 30; used in ES filter
  acl_key: string;                  // hex SHA-256; cache + audit correlation
  token_schema_version: string;     // encoding schema version for acl_tokens (e.g. "v1")
  acl_version: string;              // policy-change sequence number (e.g. "v1"); distinct from token_schema_version
  claims_hash: string;              // SHA-256 of raw claims; Redis auth-cache key
  derived_at: string;               // ISO-8601 UTC timestamp of derivation
}
```

> **HLD note**: `token_schema_version` and `acl_version` are distinct fields (see `HLDv1.0/02-authorization-bootstrap.md §5`). `token_schema_version` tracks the ACL token encoding rules; `acl_version` tracks the global policy sequence. Both are included in the `acl_key` hash and in `claims_hash` for cache fencing.

### 3.2 QueryContext

Produced by Query Understanding; passed to Routing and SecureQueryBuilder.

```typescript
interface QueryContext {
  request_id: string;
  raw_query: string;
  keywords: string[];
  topic: string | null;
  doc_type: string | null;
  time_range: { year?: number; from?: number; to?: number } | null;
  intent: "factual_lookup" | "comparison" | "policy_lookup" | "summary" | "unknown";
  risk_signal: "none" | "low" | "medium" | "high";
  expanded_queries: string[];       // rule-based variants for all tiers; LLM variants added for L0/L1 only when enabled
}
```

### 3.3 RetrievalCandidate

Returned by Elasticsearch and held by the Retrieval Orchestrator.

```typescript
interface RetrievalCandidate {
  chunk_id: string;
  doc_id: string;
  content: string;                  // full chunk text
  citation_hint: {
    path: string;                   // user-visible document reference
    page_number: number | null;
    section: string | null;
  };
  topic: string;
  doc_type: string;
  acl_key: string;                  // chunk-side fingerprint (audit only)
  sensitivity_level: number;
  retrieval_score: number;          // BM25+vector fusion or BM25-only score
  source_index: string;             // which ES index this came from
}
```

### 3.4 RankedCandidate

Returned by Reranker Service to the Query Service. Contains no ACL or content fields.

```typescript
interface RankedCandidate {
  chunk_id: string;
  rerank_score: number;             // 0.0–1.0 cross-encoder relevance
}
```

### 3.5 AuditEvent

Emitted by the Audit Emitter to Audit Elasticsearch.

```typescript
interface AuditEvent {
  event_id: string;                 // uuid-v4
  request_id: string;
  timestamp: string;                // ISO-8601 UTC
  user_id: string;
  claims_digest: string;            // SHA-256 of raw claims (not acl_key)
  acl_key: string;
  acl_version: string;
  target_indexes: string[];
  retrieved_chunk_ids: string[];
  ranked_chunk_ids: string[];
  sensitivity_levels_accessed: number[];
  model_path: string;               // e.g. "cloud_l1" | "private_l2" | "private_l3"
  authorization_decision: "allowed" | "denied" | "fail_closed";
  query_risk_signal: string;        // from QueryContext.risk_signal
  answer_returned: boolean;
  latency_ms: number;
}
```

### 3.6 IngestionJob

Passed between ingestion pipeline stages.

```typescript
interface IngestionJob {
  job_id: string;
  source_type: "pdf" | "html" | "markdown" | "wiki_export" | "db_export";
  source_uri: string;
  source_metadata: Record<string, unknown>;
  raw_content: string | null;       // null before parsing; text-based formats only
  raw_content_bytes: Uint8Array | null; // binary formats (PDF); mutually exclusive with raw_content
  parsed_sections: ParsedSection[];
  chunks: Chunk[];                  // populated by Chunker stage; empty before chunking
  sensitivity_level: number | null; // set by Risk Scanner
  acl_policy: ACLPolicy | null;     // set by ACL Binder
  stage: IngestionStage;
  created_at: string;
  updated_at: string;
}

interface Chunk {
  content: string;
  page_number: number | null;
  section: string | null;
  doc_id?: string;        // set by Metadata Enricher
  chunk_id?: string;      // set by Metadata Enricher
  vector?: number[];      // set by Embedding Client
}

type IngestionStage =
  | "connector"
  | "parser"
  | "risk_scanner"
  | "chunker"
  | "metadata_enricher"
  | "acl_binder"
  | "embedding"
  | "indexer"
  | "complete"
  | "quarantined";

interface ParsedSection {
  content: string;
  page_number: number | null;
  section: string | null;
  table_cells?: string[][];
}

interface ACLPolicy {
  allowed_groups: string[];   // raw group identifiers (e.g. "eng:infra")
  allowed_roles: string[];    // raw role identifiers (e.g. "manager"); separate from groups
  acl_tokens: string[];       // pre-computed tokens (optional; recomputed by ACL Binder if empty)
  acl_key: string;
  acl_version: string;
}
```

### 3.7 ElasticsearchChunk

The final document written to Elasticsearch by the Indexer.

```typescript
interface ElasticsearchChunk {
  doc_id: string;
  chunk_id: string;
  content: string;
  vector: number[];                 // 1536-dim for L0/L1; 1024-dim for L2/L3
  path: string;
  page_number: number | null;
  section: string | null;
  topic: string;
  doc_type: string;
  year: number | null;
  source: string;
  allowed_groups: string[];         // stored for audit replay; not used in query filter
  acl_tokens: string[];             // query-time filter field
  acl_key: string;                  // chunk-side ACL fingerprint
  acl_version: string;
  sensitivity_level: number;
  created_at: string;
  updated_at: string;
}
```

## 4. Sensitivity Tier Reference

| Level | Integer | Index Name | Embedding Model | Model Path |
|-------|---------|------------|-----------------|------------|
| L0 | 0 | `public_index` | `text-embedding-3-small` (1536d) | Enterprise gateway |
| L1 | 1 | `internal_index` | `text-embedding-3-small` (1536d) | Enterprise gateway |
| L2 | 2 | `confidential_index` | `bge-m3` (1024d, multilingual) | Private deployment |
| L3 | 3 | `restricted_index` | `bge-m3` (1024d, multilingual) | Private deployment |

**Cross-tier vector constraint**: kNN search is supported within the same dimension group (L0+L1 or L2+L3). Queries spanning L0/L1 and L2/L3 simultaneously must use BM25-only (kNN branch omitted).

## 5. ACL Token Namespace Convention

| Namespace | Format | Example |
|-----------|--------|---------|
| Group | `group:<normalized-name>` | `group:eng`, `group:infra` |
| Role | `role:<role-name>` | `role:manager`, `role:admin` |
| Clearance level | `level:<N>` | `level:2` |

Token compression removes known enterprise domain suffixes (e.g., `@company.com`). Different identifiers with the same semantic meaning map to the same token (normalization is deterministic and versioned).

## 6. Configuration Conventions

All services read configuration from environment variables and a mounted `ConfigMap`. Secret values (API keys, private certificates) are sourced from Kubernetes `Secret` objects only — never from `ConfigMap` or environment variables set in pod specs directly.

Standard configuration keys:

```yaml
# Common across all services
LOG_LEVEL: info              # debug | info | warn | error
SERVICE_NAME: query-service  # used in structured logs and traces
ENVIRONMENT: production      # local | staging | production
REQUEST_TIMEOUT_MS: 30000    # overall request timeout

# Tracing
OTEL_EXPORTER_OTLP_ENDPOINT: http://otel-collector:4317
OTEL_SERVICE_NAME: <service-name>
```

## 7. Logging Conventions

All logs must be structured JSON. Required fields:

```json
{
  "timestamp": "ISO-8601",
  "level": "info",
  "service": "query-service",
  "request_id": "uuid",
  "user_id": "uid-xxx",          // include only when relevant
  "message": "...",
  "duration_ms": 42              // for timed operations
}
```

**Never log**: `acl_tokens`, `allowed_groups`, chunk content, embedding vectors, or raw claims in plaintext. Use `claims_digest` or `acl_key` for correlation.
